# 客観的技術評価レポート: multiagent-local

対象: リポジトリのソースコード、システム構造、およびデータフローの事実確認に基づく評価

## 1. アーキテクチャの事実
本システムは、LLMエージェントの実行基盤と人間向けのUIを完全に分離した非対称アーキテクチャを採用しています。
- **バックエンド（実行基盤）**: `bin/multiagent` スクリプトを通じ、各エージェントをプレーンな `tmux` ペイン内で実行しています。エージェントはサンドボックス化されたAPIではなく、標準のUNIXシェル上でネイティブに動作します。
- **フロントエンド（UI）**: 人間向けにはtmuxの生画面ではなく、`lib/agent_index/chat_core.py` を用いてログをパースした構造化チャットUIを提供しています。

## 2. トランスポート層とプロトコルの事実
システム内のプロセス間通信（IPC）は極めて軽量に設計されています。
- **通信方式**: `bin/agent-send` スクリプトを利用し、標準入力（stdin）経由でテキストデータをやり取りします。HTTP APIやメッセージブローカーは介在しません。
- **データ構造**: 送信されたメッセージは、`.agent-index.jsonl` というJSON Lines形式のファイルに「追記（Append-only）」されるイベントソーシングの形態をとっています。
- **添付ファイル**: メッセージ内に `[Attached: path]` というプレーンテキストの規約を含めるだけで、フロントエンドが動的にリッチなプレビューを生成する仕組みです。

## 3. レジリエンス（耐障害性）の事実
異常系に対するハンドリングが、コードレベルで多数実装されています。
- **排他制御とアトミック操作**: `lib/agent_index/state_core.py` などで `fcntl.flock` によるファイルロックや、一時ファイルを用いたアトミックなリネーム処理（`os.replace`）を実装しており、データ競合による破損を防いでいます。
- **プロセスのタイムアウト管理**: `hub_core.py` での `tmux` コマンド呼び出しには厳格なタイムアウトが設定されており、ハングアップによるシステム全体のデッドロックを防止しています。
- **ログの保護**: ペインログを保存する際、サイズが急減した場合は古いログを退避させるロジックが組み込まれています。

## 4. UIおよび周辺機能の実装事実
- **依存関係**: フロントエンド（`chat_template.html`）は React や Vue といったフレームワークに依存せず、Vanilla JS と カスタムCSS変数（CSS Variables）のみで構築されています。
- **Web Pushと暗号化**: `push_core.py` において、外部サービスに依存せず、Pythonの `cryptography` ライブラリを用いてVAPIDキーペアを自己生成し、RFC8291に準拠したペイロードの暗号化（AES-GCM）を自前で実装しています。
- **3Dレンダリング**: `.obj` や `.stl` 形式の3Dモデルをブラウザ上でプレビューするため、外部ライブラリ（Three.js等）を用いず、Canvas APIと行列演算による独自のレンダリングエンジン（`file_preview_3d.py`）を実装しています。

## 5. 結論（事実に基づく評価）
本システムは、現代のマルチエージェントフレームワークが陥りがちな「過剰な抽象化」や「重厚なミドルウェアへの依存」を避け、UNIX哲学（パイプライン、テキストストリーム、プレーンなプロセス管理）に忠実に構築されています。
モジュール分割の粒度、異常系に対する防御的プログラミング、外部依存の最小化といった事実から、本リポジトリは極めて保守性が高く、実運用に耐えうる堅牢なプロダクショングレードのソフトウェア・アーキテクチャであると評価できます。
